# Historical Weather Window Validation - FRI v2

## Objective

Strict read-only audit of the realtime weather history pipeline to determine why backend logs report `(13 hari historis)` after the FRI v2 migration, and whether that historical window remains scientifically required.

No backend, frontend, feature-engineering, prediction, model, Open-Meteo request, or configuration code was modified during this audit.

## Section 1 - Log Location

The text `(13 hari historis)` is generated by the realtime router log statement.

| Item | Finding |
|------|---------|
| File | `backend/routers/realtime.py` |
| Function | `predict_realtime()` |
| Logger | `logger = logging.getLogger("backend.realtime")` |
| Statement | `logger.info("Realtime: %s FRI=%.2f %s (%d hari historis)", ...)` |
| Historical count source | `len(preceding) if preceding else 0` |

Relevant flow:

```text
history_data = _provider.get_weather_history(wilayah, days=14)
weather = history_data[-1]
preceding = history_data[:-1] if len(history_data) > 1 else None
logger.info(..., len(preceding) if preceding else 0)
```

If Open-Meteo returns 14 valid daily records, `preceding` contains 13 records, so the log reports `13 hari historis`.

## Section 2 - Open-Meteo Request Flow

Realtime endpoint request:

```text
backend/routers/realtime.py
predict_realtime()
  -> _provider.get_weather_history(wilayah, days=14)
```

Provider request construction:

```text
backend/providers/openmeteo_provider.py
today = datetime.now(WIB).date()
start_date = today - timedelta(days=days - 1)
end_date = today
```

For realtime `days=14`:

```text
start_date = today - timedelta(days=13)
end_date = today
```

Requested Open-Meteo daily fields:

```text
precipitation_sum,
relative_humidity_2m_mean,
temperature_2m_mean,
temperature_2m_max,
temperature_2m_min
```

Requested record count:

```text
14 calendar days, inclusive of today
```

The provider then iterates over `daily["time"]`, skips incomplete days, and returns valid `RawWeatherData` records sorted in Open-Meteo response order.

## Section 3 - Historical Window Diagram

Current realtime window when all 14 records are valid:

```text
Open-Meteo response, 14 records:

D-13  D-12  D-11  D-10  D-9  D-8  D-7  D-6  D-5  D-4  D-3  D-2  D-1  D0
 |     |     |     |     |    |    |    |    |    |    |    |    |    |
 history_data[0] .............................................. history_data[-1]

Router split:

preceding = history_data[:-1]
weather   = history_data[-1]

Log count:

len(preceding) = 13
```

Reason for 13 historical days:

```text
The router requests 14 records, reserves the latest record as prediction day, and logs the remaining 13 preceding records.
```

This is not caused by Open-Meteo requiring 13 historical days. It is caused by the realtime router explicitly requesting `days=14` and splitting off the latest day.

## Section 4 - Rain7 Construction

Realtime feature path:

```text
predict_realtime()
  -> predict_from_raw(weather, weather_history=preceding, model=model, top_n=top_n)
  -> effective_history = [{"rr": w.rr} for w in weather_history]
  -> build_prediction_features_v2(weather, history=effective_history)
  -> build_features_v2(raw_data, history=effective_history)
```

Rain7 code:

```text
ml/feature_engineering/builder.py
rr_values = [float(h["rr"]) for h in history] + [rr]
rr_series = pd.Series(rr_values)
rain7 = float(compute_rain7(rr_series).iloc[-1])

ml/feature_engineering/rainfall.py
compute_rain7(rr_series) = rr_series.rolling(window=7, min_periods=1).sum()
```

When 13 preceding records are provided:

```text
rr_values length = 13 historical RR values + 1 current RR = 14 values
rolling(window=7).iloc[-1] uses only the last 7 values
```

Rain7 used-record diagram:

```text
Input to build_features_v2:

D-13  D-12  D-11  D-10  D-9  D-8  D-7  D-6  D-5  D-4  D-3  D-2  D-1  D0
 unused unused unused unused unused unused unused used used used used used used used

Final Rain7 = sum(D-6, D-5, D-4, D-3, D-2, D-1, D0)
```

Because `Rain7` includes current day, the scientifically necessary full weekly window is:

```text
6 preceding rainfall records + current RR = 7 total records
```

## Section 5 - Requested, Used, And Unused Days

Assuming Open-Meteo returns all 14 requested days as valid records:

| Category | Count | Explanation |
|----------|-------|-------------|
| Requested total daily records | 14 | `days=14`, inclusive `start_date` through `end_date`. |
| Logged historical records | 13 | `history_data[:-1]`, excluding current/latest day. |
| Records passed to `build_features_v2` for rainfall series | 14 | 13 historical RR values plus current RR. |
| Records actually used by final `rolling(window=7).iloc[-1]` | 7 | Last 6 historical RR values plus current RR. |
| Historical records actually needed for FRI v2 Rain7 | 6 | D-6 through D-1. |
| Historical records unused by final Rain7 | 7 | D-13 through D-7. |

Unused historical data:

```text
7 preceding days are downloaded and converted, but do not affect the final FRI v2 feature vector.
```

The unused records are not used by `RR`, `RH_avg`, or `Tavg` either, because those features come from the latest `weather` record only.

## Section 6 - Scientific Necessity

Does FRI v2 scientifically require 13/14 days of historical weather?

```text
NO
```

Justification:

| Evidence | Interpretation |
|----------|----------------|
| FRI v2 feature set is exactly `RR`, `Rain7`, `RH_avg`, `Tavg`. | Only one historical/rolling feature remains: `Rain7`. |
| `Rain14` is explicitly removed from FRI v2 scoring and weighting. | 14-day antecedent rainfall is no longer part of the active scientific model. |
| `Rain7` uses `rolling(window=7, min_periods=1)`. | Final prediction only needs the latest 7 rainfall observations including current day. |
| `RR`, `RH_avg`, `Tavg` are current/latest-day features. | They do not consume older historical records. |
| Sprint v2.5A validation states Rain7 example uses 7 total values: 6 previous + current in final rolling window. | Confirms training methodology only requires weekly accumulation for final realtime prediction. |

FRI v2 scientifically requires a 7-day rainfall accumulation window, not a 14-day weather window.

## Section 7 - Legacy Analysis

Verdict:

```text
B - A leftover from FRI v1 implementation.
```

Evidence:

| Evidence | Interpretation |
|----------|----------------|
| Legacy v1 builder computes `rain3`, `rain7`, and `rain14`. | A 14-record window was useful for v1 `rain14`. |
| Realtime router still requests `days=14`. | The request length matches legacy `Rain14`, not FRI v2. |
| FRI v2 removed `Rain14`, `Rain3`, `TempRange`, and `RainfallAnomaly`. | The active v2 feature vector no longer needs 14 days. |
| `build_features_v2` only computes `Rain7` from history. | Extra data earlier than D-6 cannot affect final prediction. |

The 13 historical-day log remains accurate as an implementation description, but it no longer describes a scientifically required FRI v2 input window.

## Section 8 - Performance Impact

The current implementation requests and processes extra daily weather records. The impact is small but unnecessary.

| Area | Estimated Impact |
|------|------------------|
| Additional API payload | About 7 extra daily records per realtime request, each with 5 daily fields plus date. This is small in absolute size. |
| Additional computation | Minimal. The backend loops over 14 records instead of 7 and pandas computes rolling over 14 values instead of 7. |
| Memory impact | Negligible. 7 extra `RawWeatherData` objects plus temporary dict/Series entries. |
| Prediction latency impact | Likely negligible relative to geocoding and Open-Meteo network latency, but avoidable. |
| External API efficiency | Slightly less efficient than necessary; payload and parsing are roughly double what FRI v2 needs for Rain7. |

Performance conclusion:

```text
Low runtime impact, but the requested window is wider than necessary for FRI v2.
```

## Section 9 - Optimization Assessment

Would reducing requested history change prediction results?

If reduced carefully to 7 total records inclusive of current day:

```text
No, it should not change prediction results for FRI v2 when all needed records are present.
```

Reasoning:

```text
Current final Rain7 uses D-6 through D0.
Requesting D-6 through D0 directly would produce the same final rolling(window=7).iloc[-1] value.
```

Important boundary condition:

```text
The provider currently skips incomplete days. If incomplete days occur within the 7-day range, requesting exactly 7 calendar days could produce fewer than 7 valid observations. The existing 14-day request incidentally provides a buffer for missing/incomplete records, but this buffer is not documented as an intentional FRI v2 requirement.
```

Therefore:

| Optimization | Expected Result |
|--------------|-----------------|
| Request exactly 7 calendar days including current day | Same prediction when all 7 records are complete; possible lower valid-count resilience if some days are skipped. |
| Request 8-10 calendar days as a small completeness buffer | Likely same prediction in normal cases while reducing over-fetch compared with 14. |
| Keep 14 days | Stable but carries legacy over-fetch. |

## Section 10 - Recommendation

Recommendation:

```text
OPTIMIZE HISTORY WINDOW
```

Reason:

```text
FRI v2 only needs Rain7 as the historical feature. The final Rain7 value uses 7 total rainfall observations including current day. The current 14-day request and 13-day historical log are inherited from the legacy v1/Rain14 design and are not scientifically required by FRI v2.
```

This should be done in a future maintenance sprint, not in this read-only audit.

Suggested future decision points:

```text
1. Decide whether to request exactly 7 calendar days or keep a smaller buffer for skipped incomplete days.
2. Update realtime route comments/log labels to reflect FRI v2 Rain7 semantics.
3. Verify Rain7 parity before and after optimization on complete 14-day samples.
4. Preserve API response shape unless a separate API migration is approved.
```

## Final Assessment

| Question | Answer |
|----------|--------|
| Requested historical days | 13 preceding days plus current day = 14 total records requested. |
| Actually used historical days | 6 preceding days plus current day = 7 total records used for final `Rain7`. |
| Reason for `13 hari historis` | Router requests `days=14`, assigns latest record to `weather`, and logs the remaining 13 records as `preceding`. |
| Scientific necessity | NO. FRI v2 needs weekly antecedent rainfall (`Rain7`), not 13/14 days. |
| Legacy assessment | The 14-day window is a leftover from FRI v1/Rain14-era implementation. |
| Performance impact | Low but unnecessary over-fetching and processing of 7 extra daily records in normal complete-data cases. |
| Final recommendation | Historical window should be optimized in a future maintenance sprint. |

Final choice:

```text
Historical window should be optimized in a future maintenance sprint.
```
