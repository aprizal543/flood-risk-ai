# Limitations and Future Work

## Sprint KB6

### Current Limitations

#### 1. Knowledge Base Completeness
- **Descriptive (not prescriptive)**: Current KB captures expert knowledge as structured data but does not learn from outcomes
- **Scientific references pending**: Many references marked as `[Pending]` or `[Assumption]` — need domain expert validation
- **Commodity scope limited**: 17 horticultural commodities relevant to Riau Province only
- **No image/data attachments**: Knowledge records are text-only

#### 2. Rule Engine Rigidity
- **Static decision table**: 15 rules hardcoded — no runtime rule modification
- **No conditional rules**: Rules don't consider seasonality, market prices, or supply chain
- **No priority weighting**: All rules treated equally within each vulnerability/risk combination
- **No uncertainty handling**: Rules are 100% binary — no probabilistic recommendations

#### 3. Feature Flag Implementation
- **Requires restart**: `USE_KNOWLEDGE_RECOMMENDATION` is read at import time — environment change requires application restart
- **Single flag**: No per-user or per-tenant flag granularity
- **No gradual rollout**: No canary deployment or A/B testing capability

#### 4. Performance
- **No response caching**: Each request re-evaluates all 17 commodities (though this is ~0.1ms)
- **No query optimization**: All commodities fetched and evaluated — no pre-filtering

#### 5. API
- **knowledge_source not used in frontend**: The `knowledge_source` field is generated by the backend but not consumed by the frontend dashboard
- **No dedicated KB management API**: KB data is file-based, no REST endpoint for CRUD operations

#### 6. Testing
- **No load testing**: Performance tested at unit level but not under concurrent load
- **No chaos testing**: Reliability tested with single failures, not cascading failures
- **Limited API integration tests**: No actual HTTP-level end-to-end tests

### Future Improvements

#### Short-term (Next Sprint)
1. **Dynamic feature flag**: Implement admin endpoint to toggle flag without restart
2. **Frontend knowledge_source integration**: Display KB version and engine info in dashboard footer
3. **Expand validation**: Add load testing with Locust/k6 for concurrent requests
4. **API integration tests**: Add httpx-based E2E tests for prediction workflows

#### Medium-term
1. **KB Management API**: CRUD endpoints for commodity knowledge with admin authentication
2. **Rule versioning**: Support multiple rule sets with activation scheduling
3. **Market-aware recommendations**: Integrate commodity prices and market demand
4. **Seasonal adjustments**: Modify recommendations based on planting calendar (musim tanam)
5. **Feedback loop**: Track recommendation adoption rates and adjust rules accordingly

#### Long-term
1. **ML-augmented knowledge**: Hybrid approach where rules are generated/optimized from outcomes
2. **Multi-region support**: Extend commodity profiles to other Indonesian provinces
3. **Dynamic rule engine**: Support custom rules via DSL or configuration file
4. **Real-time knowledge sync**: Sync KB from database instead of file-based loading
5. **Recommendation explainability dashboard**: Web UI for exploring why specific commodities are recommended
6. **Integration with early warning systems**: Connect to BMKG flood warnings for proactive recommendations
